Identity Software Integration Patterns

This document outlines the standard identity patterns used within the MoJ to ensure services can talk to each other securely while protecting user data.

On behalf of (OBO) Pattern

Use Case

OBO flows are required when an application needs to act as a middleman. It exchanges the user’s initial login token for a new token intended for a different downstream service, all while maintaining the identity and permissions of the logged-in user.

Non-Tech Analogy

Imagine giving a lawyer “Power of Attorney” to sign a house deed for you. The lawyer doesn’t sign as themselves; they sign on your behalf using the specific authority you granted them.

Scenario A: No OBO Needed

Flow: (Single Hop) User -> (Web App -> Internal API)

If the Web App is the only service talking to the API, it can simply request an Access Token specifically for that API during the initial login. No exchange is required because the client requested the correct “key” upfront.

Diagram

sequenceDiagram
    participant User as User (Browser)
    participant Entra as Entra ID
    participant WebApp as Web App (Frontend/Client)
    participant API as Internal API (SDS)

    Note over User, WebApp: Initial Login Request
    User->>WebApp: Access Site / Login
    WebApp->>Entra: Redirect to Login<br/>(Request scope: api://internal-api/access)

    Note over User, Entra: User provides credentials & consent
    Entra-->>WebApp: Returns Auth Code
    WebApp->>Entra: Exchange Code for Access Token
    Entra-->>WebApp: Issues Access Token (Audience: Internal API)

    Note over WebApp, API: The "Single Hop" Call
    WebApp->>API: GET /files (with Access Token)
    Note right of API: Validates 'aud' = Internal API<br/>Reads 'oid' = User ID
    API-->>WebApp: Returns Data
    WebApp-->>User: Renders Content

Scenario B: OBO Required (Double Hop)

Flow: User -> Web App -> Internal API --(call)--> External/Downstream API

If an Internal API needs to call a second service (like a Shared Document API) as the user, it cannot reuse the token it was given. Tokens are “addressed” to a specific recipient (the Audience). The Internal API must trade the user’s token for a new one addressed to the downstream service.

Diagram

sequenceDiagram
    participant User as User (Browser)
    participant WebApp as Web App (Frontend)
    participant API1 as Internal API (SDS)
    participant Entra as Entra ID
    participant API2 as Shared Document API

    Note over User, WebApp: Initial Interaction
    User->>WebApp: Performs action
    WebApp->>API1: Call with Token A (Aud: Internal API)

    Note over API1, Entra: The "Double Hop" Exchange
    API1->>Entra: OBO Request:<br/>"Exchange Token A for a token for API2"
    Note right of Entra: Validates Trust & User Permissions
    Entra-->>API1: Issue Token B (Aud: Shared Document API)

    Note over API1, API2: Final Hop
    API1->>API2: Call with Token B
    Note left of API2: Validates 'aud' = API2<br/>Identifies original User
    API2-->>API1: Return Shared Documents
    API1-->>WebApp: Process & Return Data
    WebApp-->>User: Success

Terraform

These Terraform examples are not to be used for production use or implemented into our repository. They are purely for psuedocode examples and the IDAM team will ensure a module is available to achieve these setups.

  • The Web App has permission to call the Internal API.
  • The Internal API has permission to call the Shared Document API.
  • The Internal API is “pre-authorized” by the Shared Docs API, enabling the silent On-Behalf-Of (OBO) flow.
provider "azuread" {}

# ==============================================================================
# 1. RANDOM UUIDS (For Stable Scope IDs)
# ==============================================================================
resource "random_uuid" "scope_internal_api" {}  # Scope exposed by Internal API
resource "random_uuid" "scope_shared_docs" {}   # Scope exposed by Shared Docs API


# ==============================================================================
# 2. SHARED DOCUMENT API (The Final Backend)
# ==============================================================================
resource "azuread_application" "shared_docs" {
  display_name     = "Shared Document API (Backend)"
  identifier_uris  = ["api://shared-documents"]
  sign_in_audience = "AzureADMyOrg"

  api {
    oauth2_permission_scope {
      admin_consent_description  = "Allow the app to access shared documents."
      admin_consent_display_name = "Read Shared Documents"
      enabled                    = true
      id                         = random_uuid.scope_shared_docs.result
      type                       = "User"
      value                      = "Documents.Read"
    }
  }
}

resource "azuread_service_principal" "shared_docs_sp" {
  client_id = azuread_application.shared_docs.client_id
}


# ==============================================================================
# 3. INTERNAL API (The Middle Tier - SDS)
# ==============================================================================
resource "azuread_application" "internal_api" {
  display_name     = "Internal SDS API (Middle Tier)"
  identifier_uris  = ["api://internal-sds"]
  sign_in_audience = "AzureADMyOrg"

  # 3a. Expose its own Scope (for the Web App to call)
  api {
    oauth2_permission_scope {
      admin_consent_description  = "Allow the app to access SDS on behalf of the user."
      admin_consent_display_name = "Access SDS"
      enabled                    = true
      id                         = random_uuid.scope_internal_api.result
      type                       = "User"
      value                      = "user_impersonation"
    }
  }

  # 3b. Request Access to the Shared Document API (Backend)
  required_resource_access {
    resource_app_id = azuread_application.shared_docs.client_id

    resource_access {
      id   = random_uuid.scope_shared_docs.result # Requesting 'Documents.Read'
      type = "Scope"
    }
  }
}

resource "azuread_service_principal" "internal_api_sp" {
  client_id = azuread_application.internal_api.client_id
}

# 3c. Client Secret (Required for OBO flow to authenticate this API)
resource "azuread_application_password" "internal_api_secret" {
  application_id = azuread_application.internal_api.id
  display_name   = "OBO Client Secret"
}


# ==============================================================================
# 4. WEB APP (The Frontend)
# ==============================================================================
resource "azuread_application" "web_app" {
  display_name     = "Case Management Web App (Frontend)"
  sign_in_audience = "AzureADMyOrg"

  web {
    redirect_uris = ["https://localhost:5000/getAToken"]
    implicit_grant {
      access_token_issuance_enabled = false
      id_token_issuance_enabled     = true # Standard for OIDC login
    }
  }

  # 4a. Request Access to the Internal API (Middle Tier)
  required_resource_access {
    resource_app_id = azuread_application.internal_api.client_id

    resource_access {
      id   = random_uuid.scope_internal_api.result # Requesting 'user_impersonation'
      type = "Scope"
    }
  }
}

resource "azuread_service_principal" "web_app_sp" {
  client_id = azuread_application.web_app.client_id
}

resource "azuread_application_password" "web_app_secret" {
  application_id = azuread_application.web_app.id
}


# ==============================================================================
# 5. PRE-AUTHORIZATION (The Trust Links)
# ==============================================================================

# TRUST LINK 1: Shared Docs trusts Internal API
# This enables the "Double Hop" (OBO) without user consent prompt
resource "azuread_application_pre_authorized" "shared_docs_trusts_internal" {
  application_id       = azuread_application.shared_docs.id       # The Resource (Backend)
  authorized_app_id    = azuread_application.internal_api.client_id # The Client (Middle Tier)
  permission_ids       = [random_uuid.scope_shared_docs.result]   # The Scope being granted
}

# TRUST LINK 2: Internal API trusts Web App
# This allows the Web App to call the Internal API silently
resource "azuread_application_pre_authorized" "internal_trusts_webapp" {
  application_id       = azuread_application.internal_api.id      # The Resource (Middle Tier)
  authorized_app_id    = azuread_application.web_app.client_id    # The Client (Frontend)
  permission_ids       = [random_uuid.scope_internal_api.result]  # The Scope being granted
}

Zero Trust Alternative

Flow: User -> Web App (Low Privilege) -> Internal API (High Privilege)

If you want to harden application security, you can use OBO to enforce Least Privilege.

You create two separate App Registrations. The public-facing Web App only receives a “Low Privilege” token (e.g., User.Read.All).

If the Web App is compromised and the token is stolen, the attacker can only read basic profile data. They cannot perform sensitive actions.

When the user performs a sensitive action, the secure Internal API (which is not exposed to the public internet) uses OBO to exchange that low-privilege token for a high-privilege token (e.g., User.ReadWrite.All) to complete the task.

This ensures that high-privilege tokens never exist in the browser or the frontend web layer.

Service to Service (S2S) Pattern

Use Case

Use this for background tasks where no human user is present. This is for “Machine-to-Machine” communication, such as nightly backups, automated logging, or internal file synchronization.

Non-Tech Analogy

This is like a security guard having a master key to the building. They don’t need a specific tenant to let them in; they have their own ID card that identifies them as an authorized employee.

Diagram

sequenceDiagram
    participant Client as Consumer Service
    participant Entra as Entra ID
    participant SDS as SDS API (Resource)

    Note over Client, Entra: Step 1: Authentication
    Client->>Entra: Authenticate (via Managed Identity or Secret)<br/>Request token for SDS

    Note right of Entra: Validates Identity &<br/>Assigned App Roles

    Entra-->>Client: Issue Access Token<br/>(Contains Application Roles)

    Note over Client, SDS: Step 2: Access Resource
    Client->>SDS: Call API with Access Token

    Note left of SDS: Validates Token Signature<br/>Checks for 'Files.Write' Role

    SDS-->>Client: Return Success / Store File

Terraform

These Terraform examples are not to be used for production use or implemented into our repository. They are purely for psuedocode examples and the IDAM team will ensure a module is available to achieve these setups.

# ==============================================================================
# 1. RANDOM UUIDS (For Stable Role IDs)
# ==============================================================================
resource "random_uuid" "role_files_write" {}    # App Role exposed by SDS API


# ==============================================================================
# 2. RESOURCE API (The Service being called)
# ==============================================================================
resource "azuread_application" "sds_api" {
  display_name     = "Secure Document Storage (SDS)"
  identifier_uris  = ["api://sds.internal.laa"]
  sign_in_audience = "AzureADMyOrg"

  # 2a. Define the App Role (Permission)
  app_role {
    allowed_member_types = ["Application"]                # 'Application' = S2S/Machine
    description          = "Allows background services to upload files."
    display_name         = "Files.Write"
    enabled              = true
    id                   = random_uuid.role_files_write.result
    value                = "Files.Write"
  }
}

# Service Principal for the API (Required for Role Assignment)
resource "azuread_service_principal" "sds_api_sp" {
  client_id = azuread_application.sds_api.client_id
}


# ==============================================================================
# 3. CONSUMER SERVICE (The Background Worker)
# ==============================================================================
resource "azuread_application" "client_service" {
  display_name = "Nightly Backup Service"

  # 3a. Request Access to the Resource API
  required_resource_access {
    resource_app_id = azuread_application.sds_api.client_id

    resource_access {
      id   = random_uuid.role_files_write.result      # The role ID from above
      type = "Role"                                   # 'Role' = S2S/Machine
    }
  }
}

# Service Principal for the Consumer (The Identity that runs the code)
resource "azuread_service_principal" "client_service_sp" {
  client_id = azuread_application.client_service.client_id
}

# 3b. Generate the Secret (The "Password" for the machine)
resource "azuread_application_password" "client_secret" {
  application_id = azuread_application.client_service.id
}


# ==============================================================================
# 4. ROLE ASSIGNMENT (The "Admin Consent")
# ==============================================================================

# Unlike User flows, S2S flows require explicit assignment (Admin Consent).
# This resource grants the 'Files.Write' role to the Backup Service.
resource "azuread_app_role_assignment" "grant_files_write" {
  app_role_id         = random_uuid.role_files_write.result           # The Permission
  principal_object_id = azuread_service_principal.client_service_sp.object_id # The Consumer (Backup Service)
  resource_object_id  = azuread_service_principal.sds_api_sp.object_id    # The Resource (SDS API)
}

Technical Requirements

  • Grant Type: client_credentials
  • Entra Identity: Managed Identity (strongly preferred for Azure resources to avoid secret management) or App Registration with a Client Secret.
  • Permissions: Application Roles. These are assigned globally to the service identity rather than a specific user.

Which should you use?

If the action is… Use this Flow Permission Type
A background cron job cleaning up old files. S2S Application Role
A system log upload from a worker node. S2S Application Role
A user viewing their own uploaded documents. OBO Delegated Scope
An admin service acting on a specific user’s file. OBO Delegated Scope
